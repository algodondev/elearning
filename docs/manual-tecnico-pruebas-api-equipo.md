## 1. Introducción: dónde se ejecuta una API

Una API es un servicio que debe ejecutarse en un servidor y conectarse a dependencias como PostgreSQL. El consumidor no necesita conocer la configuración interna: envía solicitudes HTTPS a una URL y recibe respuestas del contrato público.

| Modalidad                             | Uso                                  | Cómo funciona                                                    | Responsable          |
| ------------------------------------- | ------------------------------------ | ---------------------------------------------------------------- | -------------------- |
| Local directo                         | Desarrollo                           | Node.js ejecuta la API y PostgreSQL se instala o conecta aparte. | Equipo técnico local |
| Local con Docker                      | Desarrollo reproducible              | Docker Compose coordina API y PostgreSQL en contenedores.        | Equipo técnico local |
| Servidor web/VPS                      | Demostración o producción controlada | Proceso o contenedor detrás de HTTPS; base local o administrada. | Infraestructura      |
| Servicio de aplicaciones/contenedores | Despliegue administrado              | La plataforma construye y publica la API en una URL.             | Plataforma + equipo  |

Docker empaqueta la aplicación y sus dependencias. En desarrollo, Docker Compose puede coordinar la API y PostgreSQL; en un servidor o plataforma de contenedores, la misma imagen puede publicarse detrás de HTTPS. El operador gestiona secretos, migraciones, respaldo, monitoreo, dominio y base de datos.

```text
Presentador / Postman / Swagger → URL HTTPS → API desplegada → PostgreSQL
```

Ejemplo local opcional para el responsable técnico: `docker compose up -d --build`. Las migraciones y datos iniciales se gestionan según `README.md`; no forman parte de los pasos de consumo.

**Punto de inicio del equipo:** recibir `BASE_URL`, credenciales y acceso a Swagger. No instalar Node.js, no administrar Docker y no conectarse directamente a PostgreSQL.

## 2. Cómo consumir la API

| Necesidad    | Indicación                                                                              |
| ------------ | --------------------------------------------------------------------------------------- |
| URL base     | Usar la entregada por el responsable, con `/api/v1` y sin barra final.                  |
| Credenciales | Una cuenta autorizada por rol; no compartir contraseñas en el Word.                     |
| Herramienta  | Swagger para explorar, Postman para colecciones o cURL para reproducibilidad.           |
| Datos        | Crear recursos con RUN_ID único o utilizar IDs de solo lectura entregados.              |
| Recuperación | Usar endpoints públicos; si no existe reversión, pedir restablecimiento al responsable. |

### 2.2 Herramientas

- **Swagger:** abrir la URL de documentación, ejecutar login, copiar `accessToken`, pulsar **Authorize** y probar operaciones. No pegar la palabra `Bearer` si la interfaz ya la agrega.
- **Postman:** importar la colección y el environment, configurar `baseUrl`, ejecutar login y guardar cada token en su variable.
- **cURL:** copiar los comandos de cada ficha, reemplazar placeholders y conservar `-i` para observar estado y headers.

## 3. Acceso común: health, login y perfil

Todos los presentadores deben completar esta secuencia antes de probar su área:

1. Ejecutar API-001 y confirmar 200.
2. Ejecutar API-002 para su cuenta asignada y copiar `accessToken` solo a la herramienta local.
3. Ejecutar API-003 y confirmar `role` y, para EMPLOYEE, `employeeId`.
4. Repetir login cuando el token expire; el MVP no ofrece refresh token.
5. Diferenciar 401 —no autenticado— de 403 —autenticado sin permiso o propiedad—.

```bash
curl -i -X POST "$BASE_URL/auth/login" \
  -H "Content-Type: application/json" \
  --data '{"email":"<CORREO_ASIGNADO>","password":"<CONTRASEÑA_ASIGNADA>"}'
```

**Seguridad:** no guardar tokens en este documento ni en capturas. Un JWT válido continúa sujeto a rol, propiedad, cuenta activa y rate limit.

## 4. Modelo funcional y orden de consumo

```text
Área + nivel → Empleado/cuenta → Curso → Módulo → Contenido
                         ↓
Evaluación → Preguntas → Publicación → Inscripción → Progreso → Intento
                                                              ↓
Reporte ← Alerta ← Certificado ← Aprobación

Ruta → Cursos ordenados → Asignación → Curso disponible → Aprobación → Siguiente curso
```

Una ruta posterior depende de IDs y estados anteriores. Por eso cada ficha declara prerrequisitos y cada recorrido debe iniciar desde un baseline acordado.

### API-001 — Comprobar disponibilidad de la API y la base de datos

| Campo                  | Detalle                                                                                                                                                                             |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Método y ruta          | `GET /api/v1/health`                                                                                                                                                                |

| Presentación           | Fase 1 · Presentador 1 / común                                                                                                                                                      |

| Comportamiento técnico | Ejecuta una consulta de disponibilidad contra PostgreSQL y devuelve el estado conjunto de aplicación y base de datos.                                                               |

| Prerrequisitos         | Conocer la URL base del ambiente autorizado.                                                                                                                                        |

| Errores documentados   | OpenAPI no declara respuestas de error adicionales para esta operación.                                                                                                             |
| Caso negativo/límite   | Probar UUID inexistente y, si existe propiedad, acceso con otro EMPLOYEE; esperar 404 o 403 sin filtración interna.                                                                 |
| Comprobación posterior | No cambia datos. Confirmar estado, esquema y alcance de los registros devueltos.                                                                                                    |
| Recuperación           | No requiere recuperación: es una operación de lectura o acceso.                                                                                                                     |

## 20. Recuperación y solución de problemas de consumo

- URL o red: comprobar dominio, HTTPS, prefijo `/api/v1`, VPN y health.
- 401: iniciar sesión nuevamente y confirmar que el header sea `Authorization: Bearer <token>`.
- 403: comprobar rol y que EMPLOYEE esté usando su propio employeeId/recurso.
- 404: verificar UUID y relación anidada; un módulo debe pertenecer al curso indicado.
- 409: leer `code` y revisar estado empresarial; no repetir a ciegas.
- 429: esperar la ventana indicada; no insistir con ejecuciones paralelas.
- 500: registrar hora, ruta, estado y request ID; no intentar revisar servidor o base de datos.
- Datos compartidos: crear RUN_ID único, archivar solo datos propios y solicitar restablecimiento si no existe reversión pública.
- Token expira: volver a ejecutar login; no existe refresh token en el MVP.

## Apéndice A — Estados HTTP

| Estado | Interpretación                                                            |
| ------ | ------------------------------------------------------------------------- |
| 200    | Consulta o actualización exitosa con cuerpo.                              |
| 201    | Creación, publicación o acción exitosa con cuerpo.                        |
| 204    | Operación exitosa sin cuerpo; no intentar interpretar JSON.               |
| 400    | Entrada inválida: tipos, UUID, rangos, campos o propiedades desconocidas. |
| 401    | No existe una identidad autenticada válida.                               |
| 403    | La identidad existe, pero su rol o propiedad no autoriza la acción.       |
| 404    | No existe el recurso o la relación anidada solicitada.                    |
| 409    | La solicitud es válida, pero una regla o estado empresarial la impide.    |
| 429    | Se superó el límite de solicitudes; esperar antes de repetir.             |
| 500    | Falla no esperada del servicio; conservar request ID y escalar.           |
